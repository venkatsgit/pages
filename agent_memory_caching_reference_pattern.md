# A Reference Pattern for Agent Memory and Caching

Most teams start with one question:

**"What is the status of my order?"**

This looks simple, but it is the perfect lens for designing a production-grade agent architecture with:

- exact cache
- semantic memory
- reusable workflows
- long-term user memory

This blog explains a **reference pattern** you can use irrespective of domain (orders, support, operations, observability) and irrespective of cache provider (self-hosted or cloud offerings).

---

## Why this pattern matters

If every user question always triggers full LLM reasoning + full backend calls, systems become:

- expensive
- slower over time
- inconsistent for repeated questions

A better pattern is:

1. Reuse exactly repeated answers when safe.
2. Reuse semantically similar answers when context matches.
3. Reuse deterministic workflows for known intents.
4. Preserve user/session context for personalization.

---

## The 4 memory layers

## 1. Exact Cache (General Cache)

Stores exact or near-exact request results.

Example key:

`intent=order_status + order_id=1234 + freshness_bucket=5m`

Best for:

- exact repeats
- very fast response (<10 ms local lookup in many setups)

## 2. Vector Store (Semantic Cache / Semantic Memory)

Stores query embeddings + response summary + metadata.

Best for:

- semantically similar phrasings:
  - "Where is my order?"
  - "Has my package shipped?"

## 3. LAS (Learned Agent Skills / Workflow Registry)

Stores reusable workflows by intent.

Example:

- `order_status`: extract order id -> call order API -> format response

Best for:

- consistent multi-step execution
- reducing planning variability

## 4. LTM (Long-Term Memory)

Stores persistent user/session context.

Example:

- `last_checked_order=5678`
- `preferred_notification=sms`

Best for:

- personalization
- state continuity across stateless HTTP calls

## Lets run through the different case hence you can understand the reference flow (Order example)

## Case 1: First-Time Query

User: "Where is my order?"

Resolved:

- intent = `order_status`
- order_id = `1234`

Hits/Misses:

- exact cache: miss
- vector store: miss
- LAS: miss

Action:

- LLM detects intent + extracts order id
- calls `order_tracking_api(1234)`
- generates response:
  - "Your order 1234 has shipped and will arrive tomorrow."

Write-back:

- exact cache
- vector store
- LAS candidate (new workflow)
- LTM (`last_checked_order=1234`)

### Case 1 — Miss/Store summary

- Miss:
  - exact cache: miss
  - vector store: miss
  - LAS: miss
  - LTM: optional miss (new session)
- Stored:
  - disk cache entry for `order_status + order_id=1234`
  - vector entry for `"Where is my order?"`
  - LAS candidate workflow for `order_status`
  - LTM with `last_checked_order=1234`

## Case 2: Semantically Similar Query

User: "Has my package shipped?"

Context:

- still about `order_id=1234`

Hits/Misses:

- vector store: hit
- exact cache: optional hit depending on canonicalization

Action:

- retrieve semantic match from Case 1
- optional freshness refresh using order API
- return response quickly

Write-back:

- optional exact cache refresh
- optional additional vector phrase node

### Case 2 — Miss/Store summary

- Miss:
  - LAS: usually not required for fast semantic reuse path
- Hit:
  - vector store: hit (semantic match)
  - exact cache: optional hit if normalized key maps to same context
- Stored:
  - optional cache refresh (new TTL)
  - optional extra vector entry for paraphrase phrase

## Case 3: Same Intent, New Context

User: "Where is my order 5678?"

Hits/Misses:

- LAS: hit (`order_status` workflow known)
- cache/vector for old order: miss (context changed)

Action:

- run known workflow with new id
- call `order_tracking_api(5678)`
- answer:
  - "Your order 5678 is in transit and arrives Friday."

Write-back:

- exact cache for order 5678
- vector entry with new metadata
- LTM update (`last_checked_order=5678`)

### Case 3 — Miss/Store summary

- Hit:
  - LAS: hit (`order_status` workflow reused)
- Miss:
  - exact cache: miss (new order id)
  - vector: miss or filtered out due to `order_id` mismatch
- Stored:
  - new disk cache entry for `order_id=5678`
  - new vector entry with metadata `order_id=5678`
  - LTM update to `last_checked_order=5678`

## Case 4: Personalization

User: "Send updates like last time."

Hits:

- LTM (`preferred_notification=sms`, `last_checked_order=5678`)
- LAS (`notification_workflow`)

Action:

- call `notification_api(order_id=5678, mode=sms)`
- answer:
  - "You will receive SMS updates for order 5678."

Write-back:

- exact cache for response
- LTM reinforcement

### Case 4 — Miss/Store summary

- Hit:
  - LTM: hit (`preferred_notification`, `last_checked_order`)
  - LAS: hit (`notification_workflow`)
- Miss:
  - exact cache/vector: optional miss for new command phrasing
- Stored:
  - disk cache entry for notification confirmation
  - LTM reinforcement (`preferred_notification=sms`)

## Case 5: New Intent

User: "Can I change my delivery address?"

Hits/Misses:

- cache: miss
- vector: miss
- LAS: miss

Action:

- LLM detects new intent (`change_delivery`)
- sequence:
  - `validate_order(5678)`
  - `update_address(5678, new_address)`
- answer:
  - "Delivery address updated for order 5678."

Write-back:

- exact cache
- vector entry
- LAS candidate for this new workflow
- LTM interaction history

### Case 5 — Miss/Store summary

- Miss:
  - exact cache: miss
  - vector store: miss
  - LAS: miss (new intent)
- Stored:
  - new disk cache entry for `change_delivery`
  - new vector entry for this new intent query
  - LAS candidate workflow for `change_delivery`
  - LTM recent-intent update

## Case-to-store matrix (quick reference)

| Case | Exact Cache | Vector Store | LAS | LTM |
|------|-------------|--------------|-----|-----|
| Case 1 (first-time) | Miss -> Store | Miss -> Store | Miss -> Store candidate | Miss/empty -> Store |
| Case 2 (semantic) | Optional hit/refresh | Hit -> optional Store | Usually bypass | Optional update |
| Case 3 (same intent, new context) | Miss (new key) -> Store | Miss/filtered -> Store | Hit -> reuse steps | Update |
| Case 4 (personalization) | Optional miss/hit | Optional | Hit | Hit -> reinforce |
| Case 5 (new intent) | Miss -> Store | Miss -> Store | Miss -> Store candidate | Update |

## Core insight: this is provider-agnostic

This is a **reference architecture pattern**, not a vendor-specific implementation.

You can apply it with:

- any LLM transport (direct provider, gateway, proxy)
- any exact cache technology
- any vector DB
- any LTM store

The shape of the pattern stays the same.

- JSON/YAML workflow registry
- DB table with versioned workflow definitions
- workflow engine metadata store

## Minimal runtime algorithm

For each request:

1. Normalize query + extract entities
2. Exact cache lookup
3. Vector similarity lookup
4. If unresolved:
   - intent detection
   - LAS workflow lookup
   - tool/API execution
   - LLM synthesis
5. Write-back to all memory layers

## Final sketch (text diagram)

```text
User Query
   |
   v
[Normalize + Entity Extract]
   |
   +--> [Exact Cache] ---- hit --> [Return]
   |
   +--> [Vector Store] --- hit --> [Reuse/Refresh -> Return]
   |
   v
[Intent + LAS Workflow]
   |
   v
[Tool/API Calls]
   |
   v
[LLM Response Synthesis]
   |
   v
[Write-back: Exact Cache + Vector + LAS + LTM]
   |
   v
[Final Response]
```

## What to remember

- This is not "just caching." It is **memory orchestration**.
- Exact cache optimizes repeated inputs.
- Vector store optimizes repeated meaning.
- LAS optimizes repeated workflows.
- LTM optimizes repeated user context.

That is why this pattern works across domains and cloud vendors.
